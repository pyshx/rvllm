updated rvLLM and synced the public repo/docs/paper/site to the current H100 results.

current direct-engine comparison against vLLM 0.19.0 on the same H100, same Qwen2.5-7B snapshot, same output len:

- N=1: 133.1 tok/s vs 165.5
- N=32: 4407.5 tok/s vs 4467.7
- N=64: 8038.0 tok/s vs 7972.1
- N=128: 13110.1 tok/s vs 13903.5

so the honest headline is:

- rvLLM is basically tied at N=32
- slightly ahead at N=64
- still behind at N=1 and N=128

the important architecture correction was that batch-1 had changed again.

the docs previously said the normal T=1 path was `CublasGemvDecode`. that was already stale.
the current default batch-1 path is the reusable batched scratch path, and that lifted N=1 from 127.9 to 133.1 tok/s without hurting high batch.

the batched side is still the same explicit hybrid policy:

- QKV / O / down on cuBLAS or cublasLt
- GateUp + SiLU on CUTLASS

and inside the stack, some parts are still materially faster than our own earlier baselines:

- Rust PTX JIT fused kernels are 2-7.5x faster than our hand-written CUDA in measured decode microbenches
- FA3 v3 + no-fallback validation was a 31-42% jump over the previous rvLLM attention path
- Zig SIMD sampling / conversion backend is faster on several hot CPU-side ops

still true: stock vLLM has the stronger low-batch decode stack overall. but rvLLM is now very close in the batched regime, and the public materials finally match the real shipped architecture instead of the previous one-step-old story.
