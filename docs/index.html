Abstract

Andy Norris (m0at) -- San Francisco, March 31 2026

rvLLM is a from-scratch Rust implementation of the vLLM inference engine for OpenAI-compatible API serving with CUDA GPU acceleration. 54 hand-written CUDA kernels, a Rust-native PTX compiler that generates fused kernels 2-7.5x faster than nvcc, FA3 v3 cp.async + split-KV attention, no-fallback kernel validation, cuBLAS algorithm autotuning, and CUDA graph replay across 35 batch sizes -- compiling to a single 16 MB static binary.

On H100 SXM 80GB serving Qwen2.5-7B f16, rvLLM achieves 12,312 tok/s at N=128 (direct engine, 0.85x vLLM). Direct engine gap: 0.82--0.96x (near parity at N=16/64). HTTP comparison: 0.67--0.88x. 20x faster startup, 31x smaller binary, 3x less CPU memory, and 5.6x tighter P95 latency. The JIT compiler generates shape-specialized fused kernels (RMSNorm+GEMV, SiLU*Mul+Down, Add+RMSNorm+GateUp) that are 2-7.5x faster than the hand-written CUDA equivalents on the same hardware.

JIT Kernel Compiler

Rust-native PTX generation -- no nvcc, no Triton, no Python

The rvLLM fusion compiler (crates/rvllm-fusion/src/ptx_emit.rs) emits PTX assembly directly from Rust at model load time. Kernels are shape-specialized with vectorized loads, warp shuffle reductions, and shared memory tiling tuned for the exact model dimensions. No external compiler dependency.

Three-Tier Kernel Stack

JIT > Hand-written CUDA > cuBLAS

  1. JIT fused kernels (fastest) -- Rust PTX emitter generates shape-specialized fused kernels at model load. 2-7.5x faster than hand-written CUDA for M=1 decode. Patterns: RMSNorm+GEMV, Add+RMSNorm+GEMV, SiLU*Mul+GEMV.
  2. 54 hand-written CUDA kernels -- fused decode (add+norm+QKV+bias, RoPE+cache, GQA attention, O-proj+gateup, silu+down), FP8 E4M3 variants, TMA async-prefetch GEMV, WGMMA tensor core GEMV, split-KV attention, INT4 W4A16 GEMV (4 fused variants, pending Rust wiring). No-fallback validation at startup ensures every kernel loads or the server fails fast.
  3. cuBLAS/cublasLt (batched M>1) -- autotuned algorithm selection with vendored type shim for cudarc 0.19. 32 candidates benchmarked per GEMM shape at startup.

Plus an LLVM NVPTX backend (experimental) -- full compiler pipeline: Fusion IR -> LLVM IR -> NVPTX -> PTX via inkwell. Same backend as Triton.

Where the Speed Comes From

Compound effect of eight reinforcing mechanisms

  1. JIT fused kernels -- compiler-generated fused norm+GEMV eliminates HBM round-trips
  2. GPU-resident argmax -- eliminates 74 MB DtoH per step
  3. CUDA graph replay -- 35 pre-captured graphs, 1.7 sec total capture time
  4. cuBLAS autotuning -- best algorithm per GEMM shape, benchmarked at startup
  5. FP16 hgemm + f16 KV cache -- tensor cores for all projections
  6. FA3 v3 cp.async + split-KV attention -- cp.async bulk copies, split-KV across thread blocks, parallel softmax, bank-conflict-free smem, V reuse across head groups
  7. No GIL, no GC -- deterministic Rust, parallel scheduling via Rayon
  8. FP8 weight support -- cublasLt FP8 GEMM halves weight bandwidth
